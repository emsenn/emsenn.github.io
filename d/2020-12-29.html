Automating the manual correction of failed automations in Emacs

One of the most loved and hated things about Emacs is that it allows you to do a single task in… almost endless ways, depending on what you want to accomplish. This leads to folk coming up with their own habits for solving certain problems, the way we all have a slightly different method of scrubbing a stubborn dish. I think part of the failure in many resources meant to educate about Emacs is that they portray the author's habits as truths. Fine enough for conveying the information clearly, but it means that the habit isn't explained, and that might be what really matters.

For example, I have a habit (that I've recently leaned into, after reading "You Reap What You Code" by Fred Hebert) of starting with what I can do manually (by-hand) with a computer, and then only once those systems are working well-enough, automating them. If you don't know this, some of my solutions seem absolutely ridiculous, but knowing this, they're more clear.

For example, my Emacsen currently has an issue, where Ivy-mode is not enabled upon start-up. At first, I was manually enabling the mode upon startup, with the following key sequence: M-x ivy-mode <ENTER>. I tried automating it entirely by changing my use-package declaration in my Emacs configuration, but that didn't work as intended.

So, I instead I added this to my configuration: 

(setq inhibit-startup-message t)
(setq inhibit-startup-screen t)

(setq initial-scratch-message
      (concat ";; Groundhog Emacs has some bugs upon startup.\n"
	      ";; Press C-x C-e to fix them.\n"
	      "(ivy-mode)"))

Now, when my Emacsen starts up, the Emacs startup screen is disabled, and instead I'm left looking at my scratch buffer, which shows the following: 

;; Groundhog Emacs has some bugs upon startup.
;; Press C-x C-e to fix them.
(ivy-mode)

Now, instead of M-x ivy-mode <ENTER> to fix the bug, I "just" have to do C-x C-e. This might seem like a lot of work to save very few keystrokes, but since this approach to solving problems (give myself manual control to correct) is a habit, changing the configuration didn't require much thought, nor did it require much more typing than fixing the problem one time.

(Edit: immediately after posting this I noticed another bug: exactly one record out of 578 has its title wrong in the sitemap generated by Org-publish. The daily record for yesterday, the 28th, insists (in the sitemap) its title is 2020-12-18. I did typo the title as such originally, but it should have been updated in the index, which I've deleted and re-generated several times! In this instance, my approach will be to figure out which automatic thing is going wrong, and why, and then see if I can't correct its input, rather than change the automation, which is probably correct, inside itself.)\

(Followup: (org-publish-remove-all-timestamps) fixes the bug above. Titles are expected to remain static, I guess. I'm going to write up a set of statements for "clean" publishing my Website from these records, so I'll make sure this line is somewhere early on in that process.)

Looking at Org-roam links and record relationships

Using Org-roam more, I more and more keenly appreciate the idea I had in my HashUp text-formatting syntax, of allowing for two kinds of links to a resource: a simple #mention of a term, and an ##indexed-mention of a term. Only the indexed mention would create a backlink. This would, I think, neatly solve my current issue when using Org-roam, where I want to create a link for every mention, without necessarily cluttering up the backlinks.

However, this might be because I am mis-anticipating my use of the backlinks, influenced by their rendering on emsenn.net. Within Emacs itself, and the backlinks showing some of the sentence around a link, a mere mention of a term can be very useful. However, on the Web, when the context is removed, a backlink is… more useful as navigation for getting back to where you were, and that's what Web browser's "Back" button is for!

I may be, on my Website, hoping that the lists that emerge through backlinking can serve as a list of useful resources on the topic backlinked, and I think that's a hope that automation will save me the manual work of explaining the relationships between the records.

Which is a silly thing to want to automate, when the whole point of my personal records catalog be so that I can determine, through my own thinking, my relationship to terms and their relationships.

All that to say, I think I'm going to continue using links almost whenever it seems like it might be useful to a future reader, and not be upset this doesn't create, on its own, a useful sense of relationship between records.

Looking at using Org-mode project indexing

So, as I wrote in a recent posting, I'm thinking about the relationship between my personal records and their rendering as my Website, emsenn.net.

One thing that's important to me is that the Website serve as a viable record catalog, and to me that means being able to find records in it without much friction. The abundance of linking within entries between records makes that easy-enough for my satisfaction, from within a record that the catalog contains, but if you're coming "into" the catalog, the only discovery mechanism available is the Records Index, which, in light of what I'm about to say, might be renamed to a Records Listing.

When I was using Org-mode to handle the serialization of my information about Teraum, my fantasy world, I made use of the Org-mode project index feature. I don't have those notes offhand, but an example entry might have looked like this: 

*** Arathel

    #+index: settlements!towns!Arathel
    #+index: Arathel County!settlements!capital!Arathel

This might be a way of establishing the type of index I find currently-lacking. The key will be declaring that the entries listed in such an index are manually indexed, and so the index can't be relied on as complete!
