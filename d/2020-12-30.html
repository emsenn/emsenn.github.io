Looking at structure editing and movement in Org-mode

Phil Hagelberg announced the currently-scheduled speakers for this year's FennelConf, and the thing he's going to be talking about is structure editing. It's something I've been thinking about, ever since reading through Stepan Parunashvili's "An Intuition for Lisp Syntax", where at the end they talk about how programming languages in the LISP family easily facilitate working with the code as a structure. (I also saw, somewhere on the Fediverse, someone post a demonstration video of an IDE they're developing that only allows for structure editing.

It's gotten me thinking that I'm not that familiar with structure editing and movement within Org-mode. (I also, at present, don't have any sort of structure stuff set up in Groundhog Emacs for working with LISP code either, but I'm also not really working intensively with any LISPs right now, either.)

So, while I wait on things this morning, I figured I'd take a look at the the Org-mode manual's chapter on document structure.

One thing I've learned just now is there's a setting to help catch "invisible edits." From the documentation on org-catch-invisible-edits: "Check if in invisible region before inserting or deleting a character."

I've set it to show-and-error for now, since that seems to be a good way to learn where the feature will speak up, but I'll probably toggle it it just "show" later; I tend to be pretty aware when I'm at my keyboard.

The next section in the chapter is about structure movement, an important step in structure editing. In order to help learn the Org-mode structure movement keystrokes, I'm writing a table in the new Org-mode structure editing record I've created.

(Why am I duplicating material in a manual I have locally? It's not just for the memorization aid of re-writing it, but that's a part! It's also because then this catalog slowly grows into documentation not of Emacs as a tool, but of my relationship with Emacs. Only the keystrokes I use myself tend to get documented here, so it's less of a reference for Emacs users and more a reference for people who want to understand my use of Emacs.)

Key sequence Command Description
C-UP (org-backward-paragraph) Go to the previous paragraph
C-c C-n (org-next-visible-heading) Go to next heading
C-c C-p (org-previous-visible-heading) Go to previous heading
C-c C-f (org-forward-heading-same-level) Go forward a heading at current depth
C-c C-b (org-backward-heading-same-level) Go backward a heading at current depth
C-c C-u (outline-up-heading) Go up to the parent depth's headline

So, looking at that table, and using my own language around this information management, I would say that the above keystrokes are for performing movement between entries in a record. That leaves a big part of structure movement unaccounted for: movement between records. For that, I use Org-roam… which looking at it, I pretty much just use one keystroke there.

Key sequence Command Description
C-c n f (org-roam-find-file) Open a record by searching titles

Which means there's a part of my structure movement I'm not yet addressing, and I can tell what it is: hyperlinks. (I'm not sure if this is strictly "structure" movement at this point; is moving to the definition of a function by hotkey structure movement?)

Key sequence Command Description
C-c C-o (org-open-at-point) Open the link under point
C-c C-x C-n (org-next-link) Move to the next link in the buffer
C-c C-x C-p (org-previous-link) Move to the previous link in the buffer

Hmm. In looking at the manual's chapter on handling links, I'm realizing that I'm actually using yet another Org subsystem here, which seems to be called "Org-mark." Org-mark seems to be used to save locations within different Org-mode files, and move between them. It's what the link handling system uses to handle a lot of its movement, so I guess I'll make up a little record for it.

Key sequence Command Description
C-c % (org-mark-ring-push) Adds the current position to the Org-mark "ring"
C-c @ (org-mark-subtree) Adds the current entry to the Org-mark "ring". (Use repeatedly to add subsequent entries.)
C-c & (org-mark-ring-goto) Go back to a position recorded on the Org-mark "ring". (Use repeatedly to keep going back.)

Let me put these together and see what I'm missing, if anything.

Here's how I merged the tables: First, I wrote a list of all the table's names:

  • org-roam-structure-movement-key-sequences-table
  • org-mode-structure-movement-key-sequences-table
  • org-mode-links-structure-movement-key-sequences-table
  • org-mark-structure-movement-key-sequences-table

And now a little Elisp to merge the tables, followed by the tables:

That's actually a pretty comprehensive look at how I move around the structure of my records. Next, I'll look at how I edit that structure, and then take a look at structure linking, and structure inclusion, two other things I do often that aren't (I think) strictly under "editing."

So, to talk about structure editing, I need to lay out my understanding and vocabulary around the structure of my records. (I say "my records" rather than "Org-mode files" because my records use a combination of Org-mode, Org-roam, my own code, and all of it comes from my way of thinking about recordkeeping.

Each record, a file on the disk, contains quite a bit: some meta-data that comes at the beginning of the files, and then one or more entries, each of which might contain many elements, from lists to code blocks to paragraphs.

Here's the table for generalized structure editing:

Key sequence Command Description
M-RET (org-meta-return) Insert a new heading, item, or row.
C-RET (org-insert-heading-respect-content) Insert a new heading at the end of this entry.
C-u C-u M-RET (org-meta-return) Insert a new heading at the end of this entry's parent.
M-S-RET (org-insert-todo-heading) Insert a new task entry at point.
C-S-RET (org-insert-todo-heading-respect-content) Insert a new task at thhe end of this entry.
M-LEFT (org-do-promote) Raise a headline to the parent level.
M-RIGHT (org-do-demote) Lower a headline to the child level.
M-UP (org-move-subtree-up) Move entry up within its current level.
M-DOWN (org-move-subtree-down) Move entry down within its current level.
C-c C-x C-w (org-cut-subtree) Remove an entry from the buffer and place it in the kill "ring"
C-c C-x C-w (org-copy-subtree Place a copy of the entry in the kill "ring"
C-c C-x C-y (org-paste-subtree) Places an entry from the kill "ring" into the current parent entry
C-c C-w (org-refile) Provides an interface for moving an entry to a new location
C-c ^ (org-sort) Sort all entries within a parent.

But there's also quite a bit for working with elements within entries, such as lists:

Key sequence Command Description
S-UP (org-shiftup) Move to the previous item
S-DOWN (org-shiftdown) Move to the next item
M-UP (org-metaup) Move the current item up
M-DOWN (org-metadown) Move the current item down
M-LEFT (org-metaleft) Increase the identation of the current item
M-RIGHT (org-metaright) Decrease the indentation of the current item
M-S-LEFT (org-shiftmetaleft) Increase the indentation of the current item and its children
M-S-RIGHT (org-shiftmetaright) Decrease the indentation of the current item and its children
C-c C-c (org-ctrl-c-ctrl-c) Reflow list and complete current item, if it has a checkbox

There's also Org-mode drawers, which I don't use too much right now.

Key sequences Command Description
C-c C-x d (org-insert-drawer) Ask for the name of a drawer and insert it, or wrap the region (if there is one) in that drawer.
C-c C-z (org-add-note) Adds a new timestamped note to the current entry's LOGBOOK drawer, making one if there isn't one.

There's one part of structure editing I won't cover here, which is my use of Org-mode capture templates to insert Org-mode blocks. Those templates change often; I'll wait until I can write the code to have that system self-document.

I mentioned links earlier, but now I realize I only use two keystrokes to work with them. Still worth recording:

Key sequence Command Description
C-c C-l (org-insert-link) Pulls up the interface to insert a link (either typed or selected from the link "ring", or edit the link under point
C-c l (org-store-link) Saves a link in in the link "ring"

Another important tool in my structure editing method is Org-mode inclusion. At the moment I type these out by hand, but I plan on adding some capture templates to make it a little easier. The Org-mode manual's chapter on inclusion is worth reading. I went into this feeling like I didn't really do much structure editing and had a very manual approach to making Org-mode files, when I knew there's a lot of possibility for automation. I realize now, I just have been doing some things with keystrokes without even realizing that I was doing it, because (for the most part) the keystrokes are easier than the thing, and it's not any more difficult to remember "alt-enter" than "a new line with the same number of asterisks as the previous one." By viewing both as just a means to a headline (and the headline, ultimately, as just a means to titling an entry, in my own conception of the information,) I guess I don't always view things as automatic forms of manual actions, but just a different means to an end.
